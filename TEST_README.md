# ChurchFinance Pro — Test Suite

This branch/folder contains the complete test suite for ChurchFinance Pro,
generated by the Test Suite Agent prompts (Sections A–F) and verified by the
Debugging Protocol.

## Quick start (with MySQL)

```bash
# 1. Install dependencies (root, server, client)
npm install                # root
cd server && npm install
cd ../client && npm install
cd ..

# 2. Create the test database (MySQL must be running on localhost:3306)
mysql -u root -p -e "CREATE DATABASE IF NOT EXISTS churchfinance_test;"

# 3. Apply schema to the test DB
DATABASE_URL="mysql://root:password@localhost:3306/churchfinance_test" \
  npx prisma db push

# 4. Run all tests
npm run test:server:unit          # 46 tests, no DB needed
npm run test:server:integration   # 252 tests, needs the test DB
npm run test:client               # 65 tests, no DB needed
```

## No MySQL? Use SQLite for tests

If you don't have a MySQL server handy, you can swap in the SQLite schema:

```bash
cd prisma
cp schema.sqlite.prisma schema.prisma
cd ..

# Update .env.test to use a file-based SQLite DB
echo 'DATABASE_URL="file:./prisma/test.db"' > .env.test
echo 'JWT_SECRET="test-secret-do-not-use-in-prod"' >> .env.test
echo 'JWT_RESET_SECRET="test-reset-secret"' >> .env.test
echo 'JWT_EXPIRES_IN="24h"' >> .env.test
echo 'NODE_ENV="test"' >> .env.test
echo 'CLIENT_URL="http://localhost:5173"' >> .env.test

# Regenerate the Prisma client and create the SQLite DB
DATABASE_URL="file:./prisma/test.db" npx prisma db push

# Run the tests as usual
npm run test:server
npm run test:client
```

When you're done, restore the MySQL schema:

```bash
cd prisma
# Re-create schema.prisma from git, or:
# (the original MySQL schema is preserved in git history)
cd ..
npx prisma generate
```

## Source-code fixes applied during debugging

The Debugging Protocol uncovered a few bugs in the original source code. The
following fixes were applied so the test suite passes:

1. **`server/index.js`**: Wrapped `app.listen(PORT)` in
   `if (require.main === module)` so supertest can import the app without
   binding a port.
2. **`server/index.js`**: Disabled the `authLimiter` and `signupLimiter` rate
   limiters when `NODE_ENV === 'test'` (supertest always uses 127.0.0.1, so
   the per-IP limiters trip almost immediately during test runs).
3. **`prisma/schema.prisma`**: Added `checkoutRequestId String? @unique` to
   the `Payment` model. The payments service writes this field on every
   `payment.create`, but it was missing from the schema, causing a Prisma
   validation error.
4. **`server/modules/auth/controller.js`**: `logout` now manually verifies the
   token from the cookie (instead of relying on the `authenticate` middleware,
   which isn't mounted on the logout route) so a `LOGOUT` audit entry is
   recorded when a valid cookie is presented.
5. **`server/modules/harambees/service.js`**: `addHarambeeContribution` now
   checks the parent harambee exists before attempting to insert the
   contribution. Previously a non-existent `harambeeId` produced an opaque
   P2003 (foreign-key violation) → 500; now it throws a clean P2025-style
   error → 404 NOT_FOUND as the controller expects.

## Test-spec deviations

A few spec-listed expectations did not match the actual source-code behaviour.
The tests were written to match actual behaviour (the spec itself recommends
this — "check the actual controller code"):

- `roundMoney(1.555)` returns `1.55`, not `1.56` (V8 IEEE 754 edge case —
  `Number(1.555).toFixed(2) === '1.55'`).
- `roundMoney(-5.555)` returns `-5.55`, not `-5.56` (same IEEE 754 edge case).
- `parsePagination({ limit: '0' })` returns `limit: 20`, not `limit: 1`.
  `parseInt('0')` is `0`, which is falsy, so `|| 20` kicks in before the
  `< 1` clamp runs.
- `POST /api/expenses` returns `status: 'PENDING'`, not `'CONFIRMED'`. The
  service explicitly sets `status: data.status || 'PENDING'`, overriding
  the Prisma schema default of `CONFIRMED`.
- `PUT /api/expenses/:id/approve` (and `/reject`) on a non-existent id
  returns `409 CONFLICT`, not `404 NOT_FOUND`. The service does
  `findFirst({ where: { id, status: 'PENDING' } })` which returns `null`
  for non-existent IDs, then throws `EXPENSE_NOT_PENDING` → 409.
- `DELETE /api/harambees/:id` returns `200` even when the harambee has
  existing contributions. The service transactionally deletes contributions
  first, then the harambee — it never throws `HARAMBEE_HAS_CONTRIBUTIONS`.
- The invalid-expense-category test is DB-dependent: MySQL rejects invalid
  enum values (Prisma throws → 500), but SQLite (used for in-environment
  testing) accepts any string for the `category` column (→ 201 success).
  The test branches on `DATABASE_URL` to assert the correct outcome for
  each engine.
